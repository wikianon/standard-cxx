Introduction

The XE Toolkit is the follow-on to the SE Toolkit, originally developed for the SunOS operating system in Sun Microsystems’ Solaris environment. XE represents a complete rewrite and shows its heritage only in the style of programs that it contains, its intended use as a toolkit, that is, a set of tools for developers to use so they can build their own applications, and its focus on operating system metrics. Other than these goals, there is absolutely no backwards compatibility to SE scripts. The programs that were part of the SE Toolkit were written in SymbEL, the interpreted language of SE. XE is written in Java. All of the client-side applications are pure Java and the RMI server that provides metrics to the clients is Java with native method support.

 

Although XE is the first product of Captive Metrics and although it is usable as a separate application suite, XE is a piece of a larger product suite. The Captive Metrics architecture is called Captivity and XE represents a collection of tools that can be used by Captivity as tactical monitoring components. The types of fast-answer-now solutions provided by SE carry forward in XE and will be used in CaptiveConsole, the central console component of Captivity.

 

Installation and Usage

The file INSTALL.txt at the top level of the installation contains all of the information regarding the installation of the software for all of the supported platforms. Reproducing the information here would be redundant and prone to inconsistency with that document.  For a complete discussion of XE and all of its components, refer to the XE User Manual.

 

Using the Tools

For those that simply want to use XE for the tools it provides, there are two steps. First, start the collector, CaptiveCollector. For instructions on starting CaptiveCollector, see the INSTALL.txt file as it contains information regarding the starting of CaptiveCollector for each platform. The collector should be started on any host that will be monitored. The client applications should be run on a workstation dedicated to heavy GUI lifting. It should have a great deal of memory and at least one very fast processor. The usage of the tools is located in the XE programs documentation.

 

Developing New Tools

The documentation for the libraries used by XE is in two places. The first is the documentation for the XE classes. The second is the documentation for the Captive Metrics libraries. The best source of information on how to develop new tools is to look at the programs. The XE objects are really quite easy to use and the majority of the development is building an attractive user interface.

Other Important Information

 

Usage and Purpose of the Counters program

 

In the TOP/bin directory is a program that is not documented with the rest of the applications. Its name is Counters and it will bring up a dialog box much like the Add Counters dialog box of the Perfmon program on Windows. This will allow you to peruse all of the available performance counters offered by CaptiveCollector on the system it is running. The usage of the program is Counters [host-name], so you can view the counters on a remote host as well as the local host. Eventually, this dialog will be built into the XEMon program which is an XE Toolkit version of the Perfmon program.

 

Also note that this program does not have the same look-and-feel as the other XE programs since it's only a test program.

 

Use of JAVA=/opt/java/bin/java for rpm Install on Linux

 

The rpm program on Linux is used to install XE on that platform. The package checks to verify that there is a Java virtual machine on the system at revision 1.5.0 or newer. This is done simply by calling java –version. If the java binary is not in the PATH then an error message will be issued and XE will not be installed. Unfortunately, putting the path of the compliant java VM within the PATH variable is not enough since rpm sets PATH to a default value. This can be circumvented in at least 2 ways.

  1. Put a symbolic link to the java binary in /usr/bin.

  2. Run rpm with the environment variable JAVA set to the path of the compliant VM, e.g. JAVA=/opt/java/bin/java rpm –i CMxe-0.1-1.i586.rpm.

Linux Installs are Platform Dependent

 

There are currently 2 install packages for Linux.

  1. CMxe-0.1-1.i586.rpm

  2. CMxe-0.1-1.x86_64.rpm

The x86_64 package can be installed on a machine that serves both 32 and 64 bit implementations of Linux. This is due to the feature of rpm to examine binaries contained in a package (at packaging time) to determine what shared objects are used by the binaries and then enforce a dependency rule on the existence of these files at installation time. Since the 64 bit Linux supports 32 bit applications, both sets of libraries are present on the system. Not so for the 32 bit version though. Only the 32 bit libraries are present. And it goes without saying that packaging Solaris binaries within the package does not work since they are ELF binaries and rpm determines what shared objects those binaries require and they are certainly not present on a Linux system at install time.

 

All this information comes with the caveat that I am a complete novice at building rpm packages and there may very well be a way around this issue. However, in the current alpha release, only Solaris provides all the binaries for serving to all supported platforms.

 

Modification of the message catalog files

 

Captive Metrics strives to keep all of its software multi-lingual from the very start. Unfortunately, we're short on folks who speak several languages. The result is that the only message catalogs included in the product are in U.S. English. We would very much like to fix this shortcoming. If you're fluent in a language other than English, or would like to provide a localized version of English, such as U.K. English, feel free to use the following files as templates and provide catalog files translated into your native (or second, as the case may be) language. The catalog files can be found in the following places where TOP is the installation directory of XE, such as /opt/CMxe.

 

            TOP/locale/<LOCALE>/LC_MESSAGES/mdr-messages.po

            TOP/locale/<LOCALE>/LC_MESSAGES/mdr-explanations.po

            TOP/locale/<LOCALE>/LC_MESSAGES/mdr-names.po

 

            within jar files in TOP/jars

                        CM.jar

                           com/capmet/metrics/cm/MessagesBundle_en_US.properties

                           com/capmet/metrics/cm/MessagesBundle.properties

                           com/capmet/metrics/jmdr/MessagesBundle_en_US.properties

                           com/capmet/metrics/jmdr/MessagesBundle.properties

                           com/capmet/metrics/rmi/MessagesBundle_en_US.properties

                           com/capmet/metrics/rmi/MessagesBundle.properties

                        XE.jar

                          com/capmet/tools/XE/MessagesBundle_en_US.properties

                          com/capmet/tools/XE/MessagesBundle.properties

 

Note that the <LOCALE> notation refers to the name of the locale as you would find in /usr/lib/locale on most Unix systems, such as Linux and Solaris. All of the code written thus far has been written with multi-byte characters sets (MBCS) in mind. That is to say that any translated message catalogs should use a UTF-8 character set. This was done this way since UTF-8 supersedes UTF-16 and UTF-32. The native libraries on Windows were compiled to use MBCS, not Unicode (which is UTF-16), so any development done on Windows should follow this lead. Note that the mdr-messages.po file is used on Unix only. Windows keeps all of the names of categories and counters in the Windows registry and the names are localized if you are using a native non-English Windows installation. Translation of these names is not necessary for Windows installations.

 

Of noted importance is this: if you modify the mdr-messages.po file to change the name of the categories and counters to your local language, translate the file with msgfmt, install these files in the locale directory in the XE installation and then restart CaptiveCollector with the LC_ALL environment variable set to the new locale, all of the names of the categories and counters will appear in the new language.  Running the Counters application will demonstrate this.
