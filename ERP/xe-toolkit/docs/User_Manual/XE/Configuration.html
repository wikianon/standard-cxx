Configuration

If you are reading this document, then The XE Toolkit is already installed on your system. The purpose of this discussion is to aid in the configuration of XE. The original installation instructions accompany the package binary which was used for installation. You can re-read INSTALL.txt for further clarification if needed.

TCP Port Number

By default, the registry service which listens for incoming requests for an RMI server listens on TCP port number 1099. If this port number conflicts with another or the system administrator wants the port number to be something other than 1099, it can be changed by editing the file TOP/etc/CaptiveCollector.properties and changing the property com.capmet.CaptiveCollector.RegistryPort. If the port number is changed from the default, it should be set to a value above 1023 which is out of the range of Well Known Ports. If this change is made, it must be made on all hosts acting as client or server so they will all communicate via the same TCP port.

Use of IPv4

The networking API contained within the Java runtime will use the default protocol stack on the host system. This may cause a conflict between clients and servers as the server may default to IPv6 and the client to IPv4. If this is the case, the API can be coerced to use IPv4 regardless of the default value. This can be accomplished by editing the file TOP/etc/CaptiveCollector.properties and setting the property java.net.preferIPv4Stack to true. Communication between client and server will be through IPv4 unless it is unavailable. If this change is made, it must be made on all hosts acting as client or server so they will all communicate via the same protocol stack.

Security

XE is a client/server architecture. The communication mechanism used between client and server is the Java RMI interface. RMI can be used without security, that is, without authentication or encryption. Without any modifications to configuration files, XE will use unsecured RMI. In environments where the intranet is deemed secure, this may be an adequate solution. For others more concerned about snooping information from the wire in an intranet or for internet applications, it may be more desirable to implement the secure RMI features of XE. Among the performance information transmitted from CaptiveCollector to a client application is the complete listing of running processes on the system which may contain sensitive information.

The simplest way to implement secure RMI in XE is to edit the file TOP/etc/CaptiveCollector.properties and modify the line containing the com.capmet.CaptiveCollector.Secure.* property and change the value to true. This will indicate that all hosts running CaptiveCollector are using secure RMI and using the same password. Multiple property lines can be specified to indicate that some hosts are secure and others are not. This is done by changing the * character at the end of the property name to the name of the host. Any host that is specified as being secure must use the same password. XE currently does not have unique passwords for each host. This may change in the future. An example of the use of these properties follows.

# host leopard is secure

com.capmet.CaptiveCollector.Secure.leopard=true

# host bobcat is secure

com.capmet.CaptiveCollector.Secure.bobcat=true

# all other hosts are not secure

com.capmet.CaptiveCollector.Secure.*=false

The default password used by the secure RMI in XE is CAPTIVITY. This password can be changed by modifying the files TOP/etc/CaptiveCollector.password and TOP/etc/Makefile. By changing the property com.capmet.CaptiveCollector.password in the CaptiveCollector.password file and the variable PASSWORD in the Makefile file to the same password, environments can add an extra level of security to their installation. Once the passwords have been changed, the Keystore, Truststore and CaptiveCollector.cer files must be remade by running the make command. If the XE installation is shared by all hosts via a network disk such as NFS, then changing the passwords in one place will suffice. If, however, the software is installed on local disks on multiple hosts, then the same modifications must be made on all of the hosts so that all clients and servers will have access to the same password.
